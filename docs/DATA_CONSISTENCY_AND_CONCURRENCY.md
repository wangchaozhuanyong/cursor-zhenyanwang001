# Data Consistency and Concurrency Governance

本文档规定数据一致性、多人并发、事务、幂等、缓存、报表、统计和修复数据的治理规则。

监控系统说明见 `docs/DATA_CONSISTENCY_MONITORING.md`。缓存失效规则见 `docs/CACHE_INVALIDATION_RULES.md`。后端架构规则见 `docs/ARCHITECTURE.md`。

## 1. 数据一致性原则

核心原则：

- 先明确 source of truth。
- 写入路径必须可追踪。
- 派生数据必须能重算或对账。
- 缓存必须有失效规则。
- 高风险数据修改必须可审计。
- 生产数据自动修复必须人工批准。

前端、本地缓存、报表、搜索索引都不能替代数据库里的主数据。

## 2. 主数据、派生数据和缓存

必须区分：

- 主数据：订单、用户、商品、库存、支付、优惠券、积分等业务真相。
- 派生数据：统计、报表、汇总、排行榜、仪表盘卡片。
- 缓存：前端缓存、React Query cache、PWA cache、Redis、CDN、浏览器缓存。
- 搜索索引：搜索建议、热词、搜索索引。

派生数据和缓存不一致时，不能随便改主数据迁就它们。应先确认主数据，再修复派生数据或失效缓存。

## 3. 写入路径必须可追踪

任何写入都要能回答：

- 谁触发。
- 触发入口是什么。
- 写入哪个模块。
- 写入哪个 service。
- 写入哪个 repository。
- 是否有事务。
- 是否有审计日志。
- 是否影响缓存、报表、搜索或通知。

后台敏感操作必须优先保留审计链路。

## 4. 幂等规则

以下场景必须考虑幂等：

- 支付回调。
- 手动支付确认。
- 订单状态变更。
- 优惠券领取和核销。
- 积分发放和扣减。
- 上传回调和媒体处理。
- 通知发送。
- 数据修复脚本。

幂等方式可以是唯一键、业务幂等键、状态机保护、条件更新、重复事件忽略、操作记录表。

## 5. 重复提交防护

前端可以禁用按钮、防抖、loading 锁定，但最终必须由后端防重复。

后端要考虑：

- 同一用户快速重复点击。
- 浏览器重试。
- 网络超时后用户再次提交。
- 第三方 webhook 重复投递。
- 后台员工重复操作。

## 6. 乐观锁和版本策略

对多人编辑、后台配置、库存、会员、活动、内容页等场景，优先考虑：

- `updatedAt` 条件更新。
- `version` 字段。
- 状态条件更新。
- 唯一约束。
- 冲突后提示用户刷新。

禁止默认使用 last-write-wins 覆盖别人刚改过的数据，除非业务明确允许。

## 7. 事务边界

事务应该在 service 层编排，repository 负责具体 SQL。controller 和 routes 不应开启业务事务。

订单、支付、库存、优惠券、积分等强一致场景必须明确事务边界。跨模块协作时，应通过目标模块公开能力或由主流程模块 service 编排。

如果需要把 `conn` 传入下游 repository 或服务，必须说明事务归属和回滚策略。

## 8. 条件更新

状态流转必须优先使用条件更新，例如“只有当前状态是 pending 才能改成 paid”。这样可以防止并发请求把状态改乱。

常见场景：

- 订单支付。
- 订单取消。
- 售后审批。
- 库存扣减。
- 优惠券核销。
- 积分扣减。
- 后台任务领取。

## 9. 唯一约束

以下场景应优先考虑数据库唯一约束或业务唯一键：

- 支付回调事件。
- 用户领取同一张券。
- 同一订单的同类结算记录。
- 同一任务重复执行。
- 管理员账号唯一字段。
- 第三方事件 ID。

只靠前端防重复不可靠。

## 10. 客服并发风险

客服或后台员工可能同时处理同一用户、同一订单、同一售后、同一退款。

必须考虑：

- 谁先处理。
- 后处理的人看到什么提示。
- 是否需要锁定或认领。
- 是否需要操作日志。
- 是否需要二次确认。

## 11. 多员工后台操作风险

多员工同时编辑后台配置、商品、库存、活动、优惠券、内容页时，必须避免互相覆盖。

规则：

- 重要配置保存前应确认最新版本。
- 列表批量操作要有明确范围。
- 删除和禁用必须有审计日志。
- 高风险操作应保留操作人和时间。

## 12. 多用户客户端风险

多个用户同时下单、抢券、支付、积分兑换时，前端体验不能替代后端一致性。

必须由后端保证：

- 库存不会超卖。
- 优惠券不会超领。
- 积分不会负数。
- 支付不会重复确认。
- 订单状态不会倒退。

## 13. Last-write-wins 风险

last-write-wins 只适合低风险偏好设置。以下场景禁止默认使用：

- 库存。
- 订单。
- 支付。
- 退款。
- 优惠券。
- 积分。
- 权限。
- 报表口径。
- 安全配置。

## 14. 冲突提示规则

发生并发冲突时，前端应提示用户：

- 当前数据已被别人修改。
- 需要刷新后重试。
- 本次操作没有生效。

禁止用“保存成功”掩盖冲突。

## 15. 后台操作审计日志

后台敏感操作必须尽量记录：

- 操作人。
- 操作时间。
- 操作对象。
- 操作前后状态。
- 请求路径。
- IP / user-agent。
- 失败原因。

审计日志不是普通 UI 文案，不能为了页面好看删掉或弱化。

## 16. 对账和监控规则

一致性监控见：

```text
docs/DATA_CONSISTENCY_MONITORING.md
```

报表口径见：

```text
docs/DATA_REPORTING_CONTRACT.md
```

对账任务应该能发现：

- 支付金额和订单金额不一致。
- 订单状态和支付状态不一致。
- 库存扣减和订单状态不一致。
- 优惠券状态和订单状态不一致。
- 积分流水和余额不一致。
- 报表汇总和明细不一致。

## 17. 修复数据前必须先审计

修复生产数据前必须先做审计：

- 问题范围。
- 影响用户。
- 影响订单或金额。
- 真实 source of truth。
- 修复 SQL 或脚本。
- 回滚方式。
- 是否需要停写。
- 是否需要通知运营或客服。

禁止没有审计就直接改生产数据。

## 18. 生产数据自动修复规则

不允许自动修改生产数据，除非人工批准并明确：

- 自动修复范围。
- 幂等保证。
- 最大影响条数。
- 日志位置。
- 回滚方式。
- 监控和告警。

低风险 derived data 可以自动重算，但也必须有日志。

## 19. 报表口径变更规则

不允许随意改变统计口径。报表口径变更必须人工确认，并同步：

- `docs/DATA_REPORTING_CONTRACT.md`。
- 后端报表测试。
- 前端展示说明。
- 历史数据处理方式。

## 20. 交易重构并发规则

订单、支付、库存、活动和物流属于高风险链路，必须遵守以下额外规则：

- `POST /api/orders` 必须使用服务端幂等表校验 `idempotency_key`，重复请求返回同一订单，请求指纹不同返回冲突。
- 支付回调以 provider event id / bill id / payment order id 做幂等，重复回调不得重复改订单、加销量、发积分、发通知。
- 定价统一由 pricing service 输出，商品页、购物车、结算页、订单快照和支付金额不能各算各的。
- 活动资格统一由 promotion rule engine 计算，结算和创建订单都必须重新校验活动时间、商品范围、SKU、会员等级、限购、库存和互斥叠加规则。
- 库存通过 inventory lock service 处理锁定、释放、确认扣减和恢复。前端和普通 controller 不得直接扣减库存。
- 未支付订单超时取消必须释放库存锁、优惠券、积分抵扣和活动库存占用。
- 物流轨迹写入 `logistics_tracks`，订单详情读取 `logistics_snapshot`。承运商异常只同步物流状态和异常说明，不能绕过订单状态机直接改履约状态。
- 后台高风险操作必须有审计日志，前端二次确认不能替代服务端权限、MFA 和状态校验。

## 21. Data Change Plan

数据任务开始前必须输出：

```text
Data Change Plan:
1. Source of truth:
2. Data affected:
3. Write path:
4. Transaction boundary:
5. Idempotency strategy:
6. Concurrency risk:
7. Cache/derived data impact:
8. Audit log impact:
9. Rollback plan:
10. Verification commands:
```

完成后必须输出：

```text
Data Consistency Report:
1. Source of truth preserved:
2. Transaction rule followed:
3. Idempotency handled:
4. Concurrency risk handled:
5. Cache/derived data handled:
6. Audit log handled:
7. Tests run:
8. Remaining risks:
```
