# 《宝塔卸载后影响修复手册》

> 卸载宝塔会**直接拿走**它自带的 Nginx / MySQL / Node / 计划任务 / 防火墙 / SSL 续签 / `www` 用户。本文按「会坏的项 → 受影响表现 → 修复命令 → 验证命令 → 回滚」逐条落地。
> 配套自动诊断脚本：`scripts/post-baota-fix.sh`（在服务器上执行）。

---

## 总体原则

1. **不要在宝塔卸载前就停掉宝塔的 Nginx/MySQL/Node**，否则线上立刻 502。请按 [03-删除执行清单](./03-删除执行清单.md) 的顺序：先迁数据 → 系统服务接管 → 验证 → 再卸载。
2. **凡是宝塔自动续签 / 自动备份 / 自动重启的能力，卸载后都要有等价的系统服务接替**（systemd / cron / acme.sh）。
3. **`/www/server/...`、`/www/wwwroot/...`、`/www/wwwlogs/...`、`/www/backup/...`** 这些目录在卸载脚本中可能被一起清掉，**所有要保留的内容必须先迁出**。

## 影响 1：Nginx 没了 → 站点 502 / 端口无人监听

| 项 | 说明 |
|----|------|
| **症状** | `ss -ltnp \| grep ':80\|:443'` 无 nginx；公网 502 / 拒绝连接 |
| **根因** | 宝塔的 nginx 在 `/www/server/nginx/sbin/nginx`，它的 systemd unit 也由宝塔创建 |

### 修复

```bash
# 1) 系统 Nginx
sudo apt update && sudo apt install -y nginx          # Debian/Ubuntu
# 或：sudo dnf install -y nginx                         # RHEL/Rocky

# 2) 启用站点
sudo cp /var/www/click-send-shop/deploy/nginx/site.prod.example.conf \
        /etc/nginx/sites-available/click-send-shop.conf
sudo nano /etc/nginx/sites-available/click-send-shop.conf   # 替换 YOUR_DOMAIN / root
sudo ln -sf /etc/nginx/sites-available/click-send-shop.conf \
            /etc/nginx/sites-enabled/click-send-shop.conf

# 3) 移除可能存在的默认占用 80 的站点
sudo rm -f /etc/nginx/sites-enabled/default

# 4) 校验并启动
sudo nginx -t
sudo systemctl enable --now nginx
sudo systemctl reload nginx
```

### 验证

```bash
sudo nginx -t
ss -ltnp | grep -E ':80|:443'
curl -I http://127.0.0.1/
curl -I -H "Host: YOUR_DOMAIN" http://127.0.0.1/
```

### 回滚

如紧急回退，把卸载前用 `sudo nginx -T > /root/nginx-dump-before.txt` 保存的旧配置粘回 `/etc/nginx/conf.d/` 或 `sites-available/` 后再 `nginx -t && reload`。

## 影响 2：MySQL 不可用 / 凭据失效

| 项 | 说明 |
|----|------|
| **症状** | 应用 `/api/health/ready` 非 200；`pm2-error.log` 命中 `ECONNREFUSED` / `ER_ACCESS_DENIED` |
| **根因** | 宝塔自带的 MySQL 服务路径在 `/www/server/mysql`，my.cnf 在 `/etc/my.cnf` 或 `/www/server/mysql/etc/my.cnf`；卸载后服务名/socket 可能变化 |

### 三种情况分别处理

#### A. 你的 MySQL 仍由宝塔管理（一并被卸了）

最稳路径：**卸载前**做全量备份（`mysqldump`），卸载后**重装系统包 MySQL**并恢复数据：

```bash
# 备份（卸载前）
mysqldump -h"$DB_HOST" -P"$DB_PORT" -u"$DB_USER" -p"$DB_PASSWORD" "$DB_NAME" \
  | gzip > /root/backup-db-$(date +%F).sql.gz

# 卸载后：装系统 MySQL
sudo apt install -y mysql-server          # Debian/Ubuntu
# 或：sudo dnf install -y mysql-server

# 初始化与登录
sudo systemctl enable --now mysql
sudo mysql_secure_installation

# 恢复
mysql -uroot -p -e "CREATE DATABASE click_send_shop DEFAULT CHARACTER SET utf8mb4;"
mysql -uroot -p -e "CREATE USER 'click_user'@'127.0.0.1' IDENTIFIED BY 'STRONG_PASSWORD';"
mysql -uroot -p -e "GRANT ALL PRIVILEGES ON click_send_shop.* TO 'click_user'@'127.0.0.1'; FLUSH PRIVILEGES;"
gunzip -c /root/backup-db-*.sql.gz | mysql -uclick_user -p click_send_shop
```

更新 `server/.env`：

```env
DB_HOST=127.0.0.1
DB_PORT=3306
DB_USER=click_user
DB_PASSWORD=STRONG_PASSWORD
DB_NAME=click_send_shop
```

#### B. 你的 MySQL 装在外部（RDS / 独立服务器）

无影响，**确认 `server/.env` 中 `DB_HOST` 不是 `127.0.0.1`** 即可。

#### C. 想保留宝塔的 MySQL 服务但不要面板

`/www/server/mysql` 的 mysqld 程序仍可独立运行。卸载脚本会删 mysqld 时：**先做 A 方案备份**，再装系统 mysql。**不建议**继续依赖 `/www/server/mysql` 路径。

### 验证

```bash
mysql -h"$DB_HOST" -P"$DB_PORT" -u"$DB_USER" -p"$DB_PASSWORD" -e "SHOW DATABASES;"
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:3001/api/health/ready    # 必须 200
```

## 影响 3：Node / npm / pm2 找不到

| 项 | 说明 |
|----|------|
| **症状** | `which node` 无；`pm2: command not found`；`gc-api` 进程没了 |
| **根因** | 宝塔常把 Node 装在 `/www/server/nodejs/v18.xx/bin/node`，全局 pm2 也在那里。卸载会带走整个目录与 PATH |

### 修复

```bash
# 1) 装系统级 Node 20（NodeSource 官方仓库）
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install -y nodejs build-essential
node -v
npm -v

# 2) 全局 pm2
sudo npm i -g pm2
pm2 -v

# 3) 重新接管 gc-api（在 server 目录下）
cd /var/www/click-send-shop/server
pm2 delete gc-api 2>/dev/null || true
pm2 start ecosystem.config.cjs --only gc-api --env production
pm2 save
pm2 startup systemd -u "$(whoami)" --hp "$HOME"
# 上一行会打印 sudo env PATH=... 命令，整段执行一遍
```

### 验证

```bash
pm2 list | grep -E 'gc-api .* online'
ss -ltnp | grep ':3001'
bash /var/www/click-send-shop/deploy/verify-pm2.sh
```

### 回滚

若 NodeSource 安装异常，可临时改用 nvm 装一份用户级 Node 解燃眉之急：

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/master/install.sh | bash
source ~/.bashrc && nvm install 20 && npm i -g pm2
```

## 影响 4：SSL 证书 / 自动续签失效

| 项 | 说明 |
|----|------|
| **症状** | 证书到期后浏览器报错；面板里的「自动续签」不再生效 |
| **根因** | 宝塔的证书可能在 `/www/server/panel/vhost/cert/` 或 `/etc/letsencrypt/`；自动续签由宝塔脚本调度 |

### 修复（推荐 certbot + systemd timer）

```bash
sudo apt install -y certbot python3-certbot-nginx        # Debian/Ubuntu
# RHEL: sudo dnf install -y certbot python3-certbot-nginx

# 复用现有证书（若存在）
ls -l /etc/letsencrypt/live/YOUR_DOMAIN/   2>/dev/null || true
ls -l /www/server/panel/vhost/cert/        2>/dev/null || true

# 重新签发（最稳）
sudo certbot --nginx -d YOUR_DOMAIN --non-interactive --agree-tos -m you@example.com

# 自动续签（apt 的 certbot 自带 systemd timer）
systemctl list-timers | grep certbot || sudo systemctl enable --now certbot.timer
sudo certbot renew --dry-run
```

如使用 acme.sh：

```bash
curl https://get.acme.sh | sh -s email=you@example.com
~/.acme.sh/acme.sh --issue --nginx -d YOUR_DOMAIN
~/.acme.sh/acme.sh --install-cert -d YOUR_DOMAIN \
  --key-file       /etc/nginx/ssl/YOUR_DOMAIN.key \
  --fullchain-file /etc/nginx/ssl/YOUR_DOMAIN.crt \
  --reloadcmd      "sudo systemctl reload nginx"
```

### 验证

```bash
curl -sI https://YOUR_DOMAIN | head -n 5
echo | openssl s_client -connect YOUR_DOMAIN:443 -servername YOUR_DOMAIN 2>/dev/null \
  | openssl x509 -noout -dates
sudo certbot renew --dry-run
```

## 影响 5：`www` 用户/组消失，文件无人能写

| 项 | 说明 |
|----|------|
| **症状** | `chown: invalid user 'www'`；上传 / 写日志报权限错；Nginx worker 改 `www-data/nginx` 后访问 `root` 路径 403 |
| **根因** | 宝塔的服务以 `www:www` 运行，被卸载脚本删除 |

### 修复

```bash
# 1) 把 /var/www 的属主改成实际用户与你 Nginx worker 一致
ps -eo user,comm | awk '/nginx/{print $1; exit}'   # 看 worker 用户：nginx 或 www-data

sudo chown -R ubuntu:ubuntu /var/www/click-send-shop      # 部署用户
sudo find /var/www/click-send-shop -type d -exec chmod 755 {} \;
sudo find /var/www/click-send-shop -type f -exec chmod 644 {} \;
sudo chmod -R u+rwX /var/www/click-send-shop/server/logs
sudo chmod 600 /var/www/click-send-shop/server/.env

# 2) 上传 / public 目录给 nginx worker 读权限
sudo chmod 755 /var/www /var/www/click-send-shop /var/www/click-send-shop/public-frontend
```

### 验证

```bash
ls -ld /var/www/click-send-shop /var/www/click-send-shop/server/logs
sudo -u www-data test -r /var/www/click-send-shop/public-frontend/index.html && echo OK
```

## 影响 6：日志路径丢失

| 项 | 说明 |
|----|------|
| **症状** | 业务日志、Nginx 访问日志被清空（卸载脚本删 `/www/wwwlogs/`） |
| **根因** | 宝塔默认日志目录在 `/www/wwwlogs/`，卸载后**整个目录可能被删** |

### 修复（卸载前先迁）

```bash
sudo mkdir -p /var/log/nginx /var/log/click-send-shop
sudo cp -a /www/wwwlogs/. /var/log/click-send-shop/old-baota-logs/ 2>/dev/null || true
```

并把 Nginx 站点配置写入：

```nginx
access_log /var/log/nginx/click-send-shop-access.log;
error_log  /var/log/nginx/click-send-shop-error.log;
```

应用日志已由 `server/ecosystem.config.cjs` 写到 `server/logs/pm2-*.log`，**与宝塔无关**。

### 验证

```bash
ls -la /var/log/nginx/ | grep click-send-shop
ls -la /var/www/click-send-shop/server/logs/
```

## 影响 7：计划任务（数据库备份、定时清理）丢失

| 项 | 说明 |
|----|------|
| **症状** | 不再产生备份；过期清理脚本不再运行 |
| **根因** | 宝塔的「计划任务」由其面板调度，卸载后**全部失效** |

### 修复（系统 crontab 接管）

```bash
sudo tee /etc/cron.d/click-send-shop-backup >/dev/null <<'EOF'
# 每天 03:30 备份数据库到 /root/backups
30 3 * * * root /var/www/click-send-shop/scripts/backup-mysql.sh /root/backups >> /var/log/click-send-shop/backup.log 2>&1
EOF
sudo chmod 644 /etc/cron.d/click-send-shop-backup
```

> 仓库已提供 `scripts/backup-mysql.sh`；按需修改它的连接信息或读 `server/.env`。

### 验证

```bash
sudo run-parts --test /etc/cron.d/ 2>/dev/null | grep click-send-shop || true
sudo systemctl status cron     # Debian/Ubuntu
# 或 systemctl status crond    # RHEL
ls -lh /root/backups/
```

## 影响 8：防火墙规则丢失

| 项 | 说明 |
|----|------|
| **症状** | 卸载后 `iptables -L` 全是 ACCEPT；面板里设的「禁止国家 IP / 端口白名单」失效 |
| **根因** | 宝塔安全规则由自身写入并维护 |

### 修复（ufw 最简）

```bash
sudo apt install -y ufw
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status verbose
```

> 云上服务器仍以**云安全组**为准，`ufw` 是操作系统侧的兜底。

## 影响 9：宝塔残留 systemd 单元 / 启动项

```bash
systemctl list-unit-files | grep -i -E 'bt|panel|aapanel' || true
sudo systemctl disable --now bt 2>/dev/null || true
sudo rm -f /etc/systemd/system/bt.service /lib/systemd/system/bt.service
sudo systemctl daemon-reload
```

## 影响 10：`PROJECT_DIR` 旧路径硬编码

| 项 | 说明 |
|----|------|
| **症状** | 老的 webhook / cron / `pm2 startup` 仍在 `/www/wwwroot/...` |
| **修复** | 在所有触发器（cron、CI Secret、面板任务）把 `PROJECT_DIR` 改为 `/var/www/click-send-shop`，包括：

```bash
# crontab 里
*/5 * * * * cd /var/www/click-send-shop && bash deploy/ci-deploy.sh

# GitHub Secret
DEPLOY_PROJECT_DIR = /var/www/click-send-shop

# pm2 startup（重新生成）
pm2 unstartup systemd
pm2 startup systemd -u "$(whoami)" --hp "$HOME"
pm2 save
```

# 一次跑完的「自动诊断 + 提示」

执行：

```bash
sudo bash /var/www/click-send-shop/scripts/post-baota-fix.sh
```

脚本会自动检查上述 10 项中**真正坏掉**的部分，并打印**该执行的具体命令**（**不会**自动 sudo 删改东西，避免误操作）。

---

# 一键「全部修复」自动执行

```bash
# 预演（不会改任何东西，先看看会做什么）
sudo DOMAIN=your-domain.com DRY_RUN=1 bash /var/www/click-send-shop/scripts/post-baota-autofix.sh

# 真正执行
sudo DOMAIN=your-domain.com bash /var/www/click-send-shop/scripts/post-baota-autofix.sh

# 同时修文件属主 + 启用 ufw（请先确认 22 端口已放行）
sudo DOMAIN=your-domain.com OWNER=ubuntu:ubuntu ENABLE_UFW=1 \
  bash /var/www/click-send-shop/scripts/post-baota-autofix.sh
```

`scripts/post-baota-autofix.sh` 会**幂等地**修以下项（重复运行无副作用）：

| # | 项 | 自动行为 |
|---|----|---------|
| 1 | 项目目录 / 基础工具 | 检测 + 装 curl/tar/git/rsync |
| 2 | Node 20 / npm / pm2 | 缺则用 NodeSource 装 |
| 3 | PM2 接管 + 开机自启 | 启 `gc-api`、生成 `pm2-*.service`、`pm2 save` |
| 4 | Nginx 安装 + 站点 | 用模板写 `/etc/nginx/sites-available/click-send-shop.conf`，替换 `YOUR_DOMAIN` |
| 5 | SSL 证书 + 续签 timer | `certbot --nginx -d $DOMAIN` + `certbot.timer` |
| 6 | 数据库备份 cron | 写 `/etc/cron.d/click-send-shop-backup`（每日 03:30） |
| 7 | `.env` 权限 / 属主 | 强制 `chmod 600`；`OWNER=user:group` 才会 `chown -R` |
| 8 | 防火墙 ufw | **必须** `ENABLE_UFW=1` 才动，避免锁掉 SSH |
| 9 | 宝塔 systemd 残留 | `disable --now` + 删 `bt.service` |
| 10 | MySQL | **不自动操作**（数据风险）；只检测连通性并打印恢复步骤 |
| 11 | 健康检查 | live/ready/Nginx :80/HTTPS 全通才算成功 |

末尾会输出：

```
DONE: <数量>    SKIP: <数量>    FAIL: <数量>
失败项：- ...
跳过项：- ...
```

> **不会**自动做的：动 MySQL 数据、改 shell PATH、强行启用防火墙、覆盖你已经存在的 nginx 站点配置。这三类必须人工。
